## Failure Patterns

### Pattern 1: State Leak on Timeout (CRITICAL)

- **Occurrences:** 1 (Run 9 — action-tracker falsely archived)
- **Root cause:** `solo-dev.sh` line 923-925 — when global timeout fires, it logs `TIMEOUT` and skips re-exec, but does NOT clean `.solo/states/{build,deploy,review}`. On manual restart, stale state files make the pipeline think the new plan is already complete.
- **Wasted:** Entire track (10 acceptance criteria, 14 tasks) archived with 0% completion
- **Fix:** Add state cleanup after timeout: `rm -f "$STATES_DIR/build" "$STATES_DIR/deploy" "$STATES_DIR/review"` at line 926 of `solo-dev.sh`

### Pattern 2: Pre-Start Instability

- **Occurrences:** 6 aborted starts (01:26–01:51) before successful pipeline
- **Root cause:** Rate limiting (CLI exit code 130) + incomplete configuration. Pipeline logs show 3 empty starts + 3 aborted starts + 1 rate-limit recovery.
- **Wasted:** ~25 min of startup overhead
- **Fix:** Add pre-flight validation in `solo-dev.sh` — check API key validity, rate limit status, config files before entering main loop.

### Pattern 3: Deploy Scope Confusion (Run 1 only)

- **Occurrences:** 1 iteration (Run 1, iter 2)
- **Root cause:** Deploy skill designed for web/cloud. Faced CLI/TUI tool, used AskUserQuestion which blocked.
- **Wasted:** 1 iteration (~3 min)
- **Fix:** Deploy skill needs project-type detection — auto-detect from Cargo.toml binary targets + CLAUDE.md keywords.
- **Note:** Deploy self-corrected for Runs 2-10 (all clean).

### Pattern 4: Valid Redo Cycle (not a failure)

- **Occurrences:** 1 redo cycle (Run 1, iters 4→5→6→7)
- **Root cause:** Review correctly identified UTF-8 slicing bug, missing README, no pre-commit hooks. All fixed in 3 min build iter.
- **Assessment:** Redo mechanism working exactly as designed. Quality catch, not waste.

### Pattern 5: Doubled Signal (cosmetic)

- **Occurrences:** 1 (Run 1, iter 4 — `<solo:redo/>` output twice)
- **Impact:** None — purely cosmetic.
